Goal

- Discovery + brief completion happens via chat
- Brief fields are read-only reflections of confirmed state

Implementation
1) Remove manual save buttons (non-negotiable)
- Ensure chat_save_buttons is ALWAYS [] before rendering
- Remove/disable any template blocks that render “Save as …” buttons (chat or brief)
- Do NOT delete /save-chat-field route yet (avoid regressions), but it must not be reachable via UI.

2) Read-only brief fields (Option B enforcement)
- Replace any editable textareas + “Save Remaining Anchors” form in IDI/FG configure view with a read-only list display.
- For missing fields show a muted placeholder:
  “[not yet provided — tell Mark]”
- No inputs, no textareas, no form submit for anchors.

3) Mark-driven extraction format (strict)
- Update Mark’s system prompt to require this ending format whenever a usable answer is provided:
  Proposed updates:
  - field: <one V1A key>
    value: <short extracted text>

  Confirmation question:
  Should I save these updates?

Rules:
- exactly ONE field per turn
- V1A keys only:
  business_problem, decision_to_support, market_geography, product_concept, target_audience, definition_useful_insight
- ≤100 words total
- “Should I save these updates?” is the only question
- No auto-saving

4) Confirmation endpoint (single-field save)
- Ensure POST /confirm-proposed-updates exists and:
  - validates user owns the draft study
  - validates field is a valid V1A key
  - saves using set_v1a_value() for mapped keys or direct update for native fields
  - inserts a Mark chat message: “Saved: <Field Label>.”
  - redirects back to configure view

5) Confirmation UI
- Detect the proposal format in Mark’s most recent message.
- When detected, render a single green button “Confirm & Save” directly under that Mark message.
- Clicking button posts to /confirm-proposed-updates and refreshes page.
