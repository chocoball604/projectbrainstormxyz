FIX 3J:
Coding Agent Prompt — Fix “Answer but No Proposal” Deadlock
Context
We have a chat driven system where Mark (an LLM agent) asks a targeted question, the user answers, and Mark is expected to emit a structured Proposed updates: block so the UI can render a Confirm & Save button.
A recurring failure mode exists:
•	Mark asks a question
•	User gives a valid, substantive answer
•	Mark responds with an acknowledgment or “wait for UI” message
•	No proposal is emitted
•	No green Confirm button appears
•	The system deadlocks and Mark loses grounding
This is not a UI bug and not a validation bug.
It is a forward progress protocol violation by Mark.
We need to fix this in three layers:
1.	Prompt level invariant (already identified)
2.	Server side detection
3.	Server side recovery (auto nudge)
Goal
Guarantee this invariant:
If Mark asks a targeted question and the user provides a valid answer for the currently allowed field, Mark must emit a proposal. Mark must never acknowledge or pause without proposing.
If Mark violates this, the server should detect it and recover automatically, without saving anything silently.
Scope of Work
✅ DO
•	Add a server side safety net that detects “answer but no proposal”
•	Auto nudge Mark to emit the missing proposal
•	Preserve all existing validation, confidence gating, and audit logs
•	Keep changes minimal and localized
❌ DO NOT
•	Do NOT auto save user text
•	Do NOT guess field/value server side
•	Do NOT weaken proposal confidence gating
•	Do NOT change UI logic
•	Do NOT add new user visible buttons

Existing Architecture (Assumed)
•	Chat messages stored with sender = user | mark
•	Mark proposals are identified by presence of "Proposed updates:"
•	Proposal parsing already exists (policy_parse_last_mark_proposal)
•	Confidence scoring and Confirm button gating already exist
•	send_chat / worker pipeline already inserts Mark responses

Required Fix — Server Side Safety Net
1️⃣ Detection Logic
Implement a check after a Mark response is produced.
Trigger ONLY IF ALL conditions are true:
1.	The previous Mark message (before the user reply):
o	Contains a ? (question mark)
2.	The user reply:
o	Is non empty
o	Is not confirmation only text (yes, ok, save, etc.)
3.	The current Mark response:
o	Does NOT contain "Proposed updates:"
This defines the illegal state:
Mark asked → user answered → Mark failed to propose
This detection must be purely structural.
Do not infer intent or meaning.

2️⃣ Recovery Behavior (Auto Nudge)
When the condition is detected:
A. Insert a system authored Mark message (NOT user visible as system)
Content (exact or very close):
It looks like you answered my question, but I didn’t propose an update yet.
Let me do that properly now.
This message:
•	Is attributed to mark
•	Is stored in chat history
•	Makes the recovery explicit and auditable
B. Immediately re invoke Mark with a forced instruction
Re call the Mark LLM once, with:
•	The same system prompt
•	The full chat history
•	PLUS an additional system instruction prepended:
SYSTEM: The user has provided a valid answer to your last question.
You must now emit a proposal for the appropriate field using the mandatory proposal format.
Do not acknowledge or ask a question. Emit a proposal only.
Important:
•	This is a single retry only
•	Do NOT loop
•	If Mark still fails, allow the failure to surface normally

3️⃣ Guardrails
Ensure the following invariants remain true:
•	✅ Proposal confidence scoring still runs
•	✅ Confirm button only appears if final_confidence ≥ threshold
•	✅ No data is saved without explicit user confirmation
•	✅ All Mark messages remain visible in chat history
•	✅ Audit logs remain intact

Where to Implement
Preferred locations (pick the cleanest):
•	Immediately after Mark’s response is written to chat_messages
•	Or inside the proposal policy evaluation block
•	Or as a post processing step before UI render
Avoid modifying:
•	LLM worker internals
•	Proposal parser logic
•	Frontend templates

Test Cases (Must Pass)
✅ Case 1 — Happy Path (No Change)
Mark asks → user answers → Mark proposes
→ No auto nudge triggered
✅ Case 2 — Bug Repro (Now Fixed)
Mark asks
User: “Urban runners aged 25–40…”
Mark: “Thanks — once the UI updates…”
→ Auto nudge triggers
→ Mark emits proposal
→ Green button appears
✅ Case 3 — Confirmation Text (No Trigger)
User reply = “yes”, “ok”, “save”
→ No auto nudge
→ Normal confirmation handling applies
✅ Case 4 — Clarification Needed
User reply = ambiguous / “not sure”
Mark asks follow up
→ No auto nudge

Definition of Done
•	The “answer but no proposal” deadlock is impossible
•	Green Confirm button reliably appears when it should
•	No silent saves occur
•	No regressions in proposal flow
•	Behavior is deterministic and auditable

Summary (for the agent)
You are not fixing Mark’s intelligence.
You are enforcing a protocol invariant.
This is a seatbelt, not an autopilot.
Implement detection → nudge → retry once → exit.
