Implementation Prompt — Survey Setup Wizard + Guardrails + Move “Supporting Panels” Below Brief So Far (Low Lift)
Objective
Redesign the Survey Setup section (shown immediately after Step 1 framing + selecting “synthetic_survey”) to better help users build high quality surveys without adding AI authorship. The redesign must:
1.	Increase the maximum allowed survey respondent count from 100 to 500
2.	Convert question entry into a wizard-like, intent first flow (question-by-question) with simple guardrails (non AI, deterministic). 
3.	Move Grounding Documents, Optional Context, and Personas sections below “Brief So Far”, so they don’t influence survey question writing. 
4.	Keep engineering lift minimal: no schema changes, no endpoint changes, and reuse existing survey config + question saving behavior. 
This is UI/workflow only. Do not modify Mark prompts, Ben QA logic, telemetry schemas, persona generation, or study execution.
Current Page (baseline)
The current Survey Setup page includes:
•	Core Framing card (Problem + Decision) and editable required anchors (Market/Geo, Product/Concept, Target Audience).  
•	Survey Setup config (respondent count + question count) and a long list of question blocks Q1…Qn with Save per question.
•	“Brief So Far” and precheck checklist. 
•	Grounding Documents, Optional Context, Personas (not applicable), and Mark consistency helper currently appear above/around Survey Setup in the page flow.
Scope (IN)

This is a limit increase only, based on updated cost assumptions.

B) Survey Setup redesign: “Question-by-question wizard” (minimal lift)
Goal: Replace the “wall of Q1…Qn blocks” with a guided, intent first question entry flow while still using the existing question storage and Save per question behavior.
B1) Add a “Survey framing strip” above Survey Questions
Immediately above the Survey Questions section, render a small reminder box:
Title: “Survey purpose”
Body: “Each question should help reduce uncertainty for this decision:”
Then display the current Decision to Support (from Core Framing). 
No new data fields; this is just display.
B2) Replace default question type behavior
Currently the UI shows many questions defaulting to open-ended blocks (Q1..Qn). 
Change the default so that new questions start unconfigured (no type chosen yet) and the user must choose an “intent” first.
For each question slot (Q1..Qn), show collapsed card by default:
•	“Q1 — Not configured”
•	Button: “Configure Q1”
When clicked, expand a 3-step mini wizard:
Step 1 — “What signal do you want?”
Radio buttons:
•	“Measure prevalence or intensity”
•	“Compare options or trade-offs”
•	“Understand reasons or motivations”
Step 2 — “Choose question format”
Show formats based on Step 1 selection (low lift mapping):
•	Measure prevalence/intensity → Likert, Range/Scale, Multiple choice
•	Compare → A/B Choice, A/B Image, Multiple choice
•	Understand reasons → Open-ended (with the existing max words rule), Multiple choice
Step 3 — “Write the question”
Show the existing input controls for that chosen type (reuse current fields, do not change request shapes).
Save behavior Keep “Save Q1” exactly as it is today (same endpoint + payload).
Important: This wizard must be UI-only; do not change storage schema or save endpoints.
B3) Guardrails (deterministic, non-blocking, minimal)
Add inline, non-blocking warnings under the question text box (or just above Save) using simple checks. These warnings should not prevent saving; they are guidance only.
Guardrail 1 — Possible double-barreled If question contains “ and ” / “ or ” and has more than one clause (simple heuristic), show:
•	“⚠ This question may be asking about more than one thing. Consider splitting it.”
Guardrail 2 — Leading / loaded language If question includes words like “helpful”, “improved”, “best”, “responsible”, show:
•	“⚠ This wording may influence responses. Try to keep it neutral.”
Guardrail 3 — Too short / vague If question length < N characters (choose a reasonable threshold), show:
•	“⚠ This is very short. Make sure it’s clear what you’re measuring.”
Guardrail 4 — Open-ended caution If question type is Open-ended, show:
•	“Tip: Use open-ended questions sparingly. They’re best for ‘why’, not ‘how many’.”
No rewriting. No AI. These are just warnings.
B4) Reduce overwhelm: show one active question at a time
To minimize cognitive overload, default UI should:
•	Expand only one question card at a time
•	Collapsing others automatically when a new card is opened
This can be done with simple JS and no backend change.
C) Brief So Far stays the gating view
Keep the existing “Brief So Far / Required to run this study / Precheck” block and logic unchanged. 
But place it above the moved “supporting panels” so survey writing remains the focus.
