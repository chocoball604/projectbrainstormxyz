2.1 Parsing
Take the existing _parse_proposed_update(chat_messages) functionality and move it into:
policy_parse_last_mark_proposal(chat_messages)

- Keep semantics: parse ONLY from the last Mark message.
- Require “Should I save these updates?” to exist.
- Extract field and value (robust line-based parse; no single fragile regex).
- Optionally extract model_confidence if “confidence: high|medium|low” exists.
Return Proposal dict with field/value/model_confidence set.

2.2 Scoring + gating
Move confidence scoring logic into:
policy_score_proposal(chat_messages, proposal, study_dict)

- Compute deterministic server score (as currently designed in your latest plan).
- Map model confidence to numeric if present.
- Combine conservatively: final = min(server, model_num) if model present else final = server
- Threshold: allow_confirm = final >= 0.70
- If blocked, set block_reason to a short string like:
  "low_confidence" or "format_invalid" or "ambiguous_value"

Add a single log line (flush=True) inside policy_score_proposal whenever a proposal is parsed:
PROPOSAL_CONFIDENCE study=<id> field=<field> server=<0.xx> model=<high|medium|low|none> final=<0.xx> allow=<true|false>

2.3 Validation + apply-save
Move field/value validation currently duplicated in /confirm-proposed-updates into:
policy_validate_for_save(field,value)

- Validate field is in _VALID_V1A_FIELDS
- Validate value non-empty
- Validate length sane (e.g. <=2000 chars)
Return ok/err.

Move save logic into:
policy_apply_save(conn, study_id, field, value)
- Use set_v1a_value(conn, study_id, field, value) for mapped keys
- Use V1A_LABELS for save label
Return save label.
