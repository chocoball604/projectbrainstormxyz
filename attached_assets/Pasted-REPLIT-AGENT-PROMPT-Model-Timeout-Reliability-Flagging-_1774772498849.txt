PART A — MODEL TIMEOUT + RELIABILITY FLAGGING (NO AUTO-SWITCH)
--------------------------------------------

A1) Implement a hard timeout in mark_reply_worker.py (must not crash Flask)
Problem: Slow models (e.g., gpt-5.4-pro) can exceed the OpenAI client timeout or hang, leaving “⏳ Mark is thinking…” forever and degrading UX.
We need a deterministic timeout + graceful failure path.

Requirements:
1) Add a hard wall-clock timeout in mark_reply_worker.py for the Mark LLM call.
   - Use 45 seconds (or 50 max, but must be less than the 60s client timeout).
   - If timeout occurs:
     a) Worker must NOT crash the Flask app
     b) Worker must update the placeholder message in chat_messages to a clear error message
     c) Worker must exit cleanly (return code 0 is fine)

2) The error message shown to the user MUST NOT claim model switching.
   Use EXACT copy below (verbatim):
   "Mark couldn't respond in time using the currently selected model. This request was not completed. An admin has been notified. Please try again or ask your admin to select a faster model."

3) Always replace the placeholder:
   - The chat message row with message_text == "⏳ Mark is thinking..." must be updated to either:
     - the actual Mark response, OR
     - the timeout error message above, OR
     - a generic safe failure message if something unexpected happens

4) The worker MUST NOT leave dangling DB connections.
   - Use try/finally to close sqlite3 connection(s).
   - Never hold DB open while performing the LLM call.

A2) Record an “incident” without schema changes (use existing tables)
We need immediate admin visibility and weekly QA report inclusion, without schema changes.

Implement incident logging using existing tables:
- model_health_status (already exists)
- model_health_checks (already exists)

On timeout in mark_reply_worker.py:
1) UPDATE model_health_status for that model_id:
   - status = "degraded"
   - last_tested_at = current UTC string
   - last_error = a short message including: "mark_chat_timeout" and the study_id
   (Do NOT invent new columns; use existing fields.)

2) INSERT a row into model_health_checks:
   - run_id: new random token
   - run_type: "incident_mark_chat_timeout"
   - started_at: UTC timestamp
   - finished_at: UTC timestamp
   - integration_mode: "live_calls_enabled" (or "error" if unknown)
   - summary_status: "fail"
   - details_json: JSON string containing:
       { "incident": "mark_chat_timeout",
         "model_id": "<model_id>",
         "study_id": <study_id>,
         "timeout_seconds": 45,
         "context": "send_chat/mark_reply_worker"
       }

This achieves: “admin immediately flagged” + auditable record, without schema changes.

A3) Surface reliability issues in Weekly QA Report (generate_weekly_qa_report)
V1A expects trust/QA discipline and calls out QA flow hardening. [1](https://onedrive.live.com/personal/bfec441e4a44d47e/_layouts/15/doc.aspx?resid=d9b62be3-cd0d-428f-b94a-7c631329d807&cid=bfec441e4a44d47e)
We must ensure reliability incidents are visible in the weekly QA report.

Modify generate_weekly_qa_report() in app.py:
1) Add a new section titled exactly:
   "## Model Reliability Issues"

2) Query model_health_checks for the current week window (same week_start logic already used):
   - rows where run_type == "incident_mark_chat_timeout"
   - group by model_id (parse details_json)
   - count incidents per model
   - include the latest timestamp and last_error (if available)

3) Append lines like:
   - "Model <model_id>: <count> Mark chat timeouts this week (latest: <timestamp>)"

4) If there are no incidents:
   - Add: "No reliability incidents recorded this week."

IMPORTANT: Do not change how the report is stored; keep inserting into weekly_qa_reports as it currently does.

A4) Do NOT implement model auto-switch
This is explicit:
- Do NOT change mark_model automatically.
- Do NOT attempt to “retry with a faster model”.
- Do NOT claim that the system “switched to a faster model.”
Only notify admin + instruct user to ask admin.

V1A change log explicitly includes Optional Context Prompts:

We will add an optional section that:
- does not affect precheck
- does not affect run gating
- persists per study
- does NOT require DB schema changes

B1) Storage approach (NO schema change)
Store optional context in studies.survey_brief as JSON for ALL study types.
This column already exists and is already parsed into configure_study["survey_brief_data"] in the index route. (Do not create new columns.)

Rules:
- For survey studies, survey_brief is already used.
  - Add optional context keys nested under e.g. "optional_context": {...} (do not break existing keys).
- For IDI/FG studies, survey_brief may be null today.
  - Create/overwrite survey_brief as JSON containing only optional_context (and leave other keys absent).
- Never overwrite existing survey_brief keys; always merge.

B2) Add UI section in templates/index.html
In the Study Configure panel (where anchors are shown), add a new section:

Header:
"Optional context (optional — does not affect Precheck)"

Fields (all textarea or input):
1) Competitive context (optional)
2) Cultural sensitivities (optional)
3) Adoption barriers (optional)
4) Risk tolerance (optional)

Helper text (exact):
"Optional context to enrich Mark/Lisa’s work. These fields never block execution and do not change Precheck."

Button:
"Save Optional Context"

Pre-fill:
- If configure_study["survey_brief_data"].get("optional_context") exists, populate the fields.

B3) Add a save endpoint in app.py
Add a POST route:
POST /save-optional-context/<int:study_id>

Behavior:
1) Validate user is active and owns the draft study (same pattern as other save routes).
2) Read the 4 optional values from form.
3) Load existing study["survey_brief"] JSON if present; else start {}.
4) Merge:
   data["optional_context"] = {
      "competitive_context": <str>,
      "cultural_sensitivities": <str>,
      "adoption_barriers": <str>,
      "risk_tolerance": <str>
   }
5) Write back:
   UPDATE studies SET survey_brief = json.dumps(data) WHERE id = ?
6) Redirect back to configure view.

Do NOT add any optional fields to ben_precheck failures.
Precheck remains driven ONLY by required anchors + persona count (as currently shown in UI). [2](https://onedrive.live.com?cid=BFEC441E4A44D47E&id=BFEC441E4A44D47E!sde171a4ca17d46ba9a80c6df20ef363a)
