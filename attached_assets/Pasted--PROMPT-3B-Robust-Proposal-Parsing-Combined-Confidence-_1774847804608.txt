PART 1 — Robust Proposal Parsing (Replace Regex with Line Parser)
==========================================================

1.1 Replace _parse_proposed_update(chat_messages) implementation with a robust line-based parser.

Requirements:
- Only parse the most recent Mark message (sender == "mark").
- Must require the string “Should I save these updates?” somewhere in the message.
- Must extract:
  - field: <key> from a line containing “field:” (allow leading spaces and “- field:”)
  - value: <text> from either:
      a) a line containing “value:” OR
      b) the next indented line after “field:” if it begins with “value:”
    Value may span multiple lines until a stop marker.
- Stop markers (stop capturing value when you hit any of these lines):
  - “Confirmation question:”
  - “Should I save these updates?”
  - end of message

- Add OPTIONAL extraction of model confidence:
  - A line containing “confidence:” with one of: high, medium, low
  - This line may appear inside Proposed updates block (preferred)
  - If absent, model_confidence = None (do not fail parsing)

Return dict:
{
  "field": <field>,
  "value": <value>,
  "model_confidence": <"high"|"medium"|"low"|None>
}
Return None if invalid.

Field validation:
- field must be one of existing _VALID_V1A_FIELDS (already defined).
- value must be non-empty after stripping.

IMPORTANT: Do NOT rely on a single regex. This must handle whitespace and minor formatting drift.

PART 3 — Server-side confidence scoring (deterministic)
==========================================================

3.1 Add a helper function in app.py:
compute_server_confidence(chat_messages, proposed_update, study_dict) -> float in [0,1]

Use deterministic heuristics ONLY (no LLM calls).
Recommended scoring:

Start score = 0.40

A) Prior-question alignment (strong signal)
- Find the most recent Mark message before the last user message (or just second-to-last Mark message).
- If that message clearly asked for the proposed field (match patterns), add +0.35
Patterns per field (simple keywords):
- business_problem: ["business problem", "problem are you trying", "challenge"]
- decision_to_support: ["decision", "inform", "support"]
- market_geography: ["market / geography", "market", "geography", "region", "country", "city"]
- product_concept: ["product / concept", "product", "service", "concept", "offering"]
- target_audience: ["target audience", "who is the audience", "customer segment"]
- definition_useful_insight: ["useful insight", "definition", "what would a useful insight"]

B) Value quality (medium signal)
- If len(value) between 3 and 180 chars: +0.15
- If value is one line (no newline) OR <=2 lines: +0.05

C) Uncertainty penalty (strong negative)
If value contains any of: ["not sure", "i don't know", "maybe", "tbd", "depends", "unsure"]:
- subtract 0.40

D) Multi-topic penalty (negative)
If value contains multiple-topic connectors indicating it likely spans >1 field:
[" and ", " also ", " plus ", " as well as ", ";", " / ", " then "]
- subtract 0.20

Clamp score to [0.0, 1.0] at the end.

3.2 Add a helper mapping model confidence → numeric
model_confidence_to_num:
- high -> 0.90
- medium -> 0.70
- low -> 0.40
- None -> None

PART 4 — Combine server score + model confidence (conservative)
==========================================================

4.1 Compute final_confidence as follows:
- If model_confidence_num is present:
    final = min(server_score, model_confidence_num)
  (Conservative: user should not confirm if either side is low.)
- If model_confidence_num is missing:
    final = server_score
  (We still allow confirm based on server score.)

4.2 Gate threshold:
- Confirm & Save button appears ONLY if final_confidence >= 0.70

4.3 If proposed_update exists but final_confidence < 0.70:
- Do NOT show Confirm button.
- Instead show a small neutral hint under Mark’s message (or in the brief panel):
  “Mark needs one clarification before saving this. Please respond in chat.”

Do NOT create new endpoints or UI flows; just hide Confirm.
