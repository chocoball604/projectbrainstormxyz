PART 1 — Make proposal parsing robust (REQUIRED)
=========================================================

In app.py, improve policy_parse_last_mark_proposal(chat_messages):

Current behavior:
- Requires exact “Should I save these updates?”
- Extracts value only if it appears on a separate line starting with “value:”

New required behavior:
- Still require the confirmation marker to treat it as a proposal, BUT:
  - Accept “Proposed updates:” proposals that have field/value/confidence inline on one line:
    Example: “- field: business_problem value: ... confidence: high”
  - Accept “field:” without hyphen.
  - If confidence is missing, set model_confidence = None (do not fail parse).
  - If value is missing but message includes a sentence right after “value:”, capture it even if inline.

Implementation approach:
- When encountering a line with “field:”, parse the remainder for optional “value:” and “confidence:”
- Use simple string splits:
  - After parsing field, search the same line for “value:” and capture until “confidence:” or end of line.
  - Search same line for “confidence:” and map if in {high, medium, low}.
- Preserve multiline value behavior already implemented.

Result:
- proposal dict always includes:
  field, value, model_confidence (optional)
